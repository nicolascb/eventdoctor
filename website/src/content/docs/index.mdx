## Why EventDoctor?

In event-driven architectures, documentation drifts from reality fast. Schemas change, new consumers appear, topics multiply — and suddenly nobody knows what's connected to what. **EventDoctor** closes that gap.

<CardGrid stagger>
	<Card title="Living Catalog" icon="document">
		Centralizes `eventdoctor.yaml` specs from every service into a single source of truth. No more stale wiki pages.
	</Card>
	<Card title="Documentation Automation" icon="rocket">
		Generates up-to-date documentation for events, schemas, and relationships automatically from your specs.
	</Card>
	<Card title="Validation & Governance" icon="approve-check">
		Enforces ownership, semantic versioning, and schema rules in your CI/CD pipeline — before bad specs reach production.
	</Card>
	<Card title="Monitoring Hooks" icon="warning">
		Surfaces orphaned, unconsumed, or undocumented events by connecting to your message broker in real-time.
	</Card>

## How It Works

Each service in your architecture maintains an `eventdoctor.yaml` that declares the events it produces and consumes. The CLI validates and publishes those specs to the EventDoctor API. The Web UI and the CLI itself let you explore the full event mesh — topics, schemas, producers, consumers — in one place.
